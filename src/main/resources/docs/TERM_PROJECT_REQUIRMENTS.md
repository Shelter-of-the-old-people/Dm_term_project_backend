|------|------|

## 2. 사용자 역할 (Role)

### 2.1 개발자
- 의뢰인이 등록한 프로젝트를 검토하고, 자신의 기술과 경험을 기반으로 프로젝트 수행을 제안하는 사용자

- 프로젝트를 등록하고 개발자에게 업무를 의뢰하는 사용자

> **참고**: 개발자, 의뢰인 모두 **회원가입 기능은 생략**하고, 검사 시 사전에 계정을 만들어둔다.

## 3. 개발자(Developer) 요구사항

### 3.1 마이페이지 - 프로필 수정 `[기능 3, 4번]`

#### 3.1.1 프로필 등록 (평가 제외)
- 등록 기능은 평가에서 제외되며, 수정 기능만 구현하면 됨

#### 3.1.2 수정 가능 항목
좌측 정보가 수정 가능해야 함:

| 항목 | 형태 | 비고 |
|------|------|------|
| **프로필 이미지** | 파일 업로드 | 별도 처리 방식 명시 (아래 3.1.4 참고) |
| **지원분야** | 체크박스 (다중) | 개발 / 디자인 / 기획 |
| **활동가능여부** | 체크박스 | 가능 시 체크 |
| **상주가능여부** | 체크박스 | 가능 시 체크 |
| **지역** | 셀렉트 박스 (도/시 2단계) | 예: 경상북도 - 구미시 |
| **형태** | 셀렉트 박스 + 경력연차 | 개인프리랜서 / 팀프리랜서 / 개인사업자 / 법인사업자 |
| **검색태그** | 태그 입력 | 아래 3.1.3 규칙 적용 |
| **소개글** | 텍스트 영역 | 자유 입력 |

#### 3.1.3 검색태그 규칙 `[기능 3번 핵심]`
- **x 버튼**을 눌러 태그 삭제 가능해야 함
- **5개 이상** 등록 시 등록될 수 없음을 사용자에게 알려야 함
- 단, 추가 버튼/입력창 등 등록 구현 편의를 위해 **UI는 자유 변경 가능**

#### 3.1.4 프로필 이미지 업데이트 `[기능 4번 핵심]`
다음 처리 흐름을 반드시 준수해야 함:

```
1. 클라이언트에서 프로필 이미지 선택
2. multipart/form-data 방식으로 서버에 전송
3. 서버가 이미지 수신
4. 서버 로컬 폴더에 저장
5. DB에는 이미지 파일 자체가 아닌 "저장 경로" 또는 "파일명"만 저장
6. 화면에서는 그 경로를 이용해 이미지 출력
```

> **검사 방식**: DB에 해당 경로가 들어갔는지 + 로컬 디렉토리에 파일이 실제로 존재하는지 검사
> **금지 사항**: 클라이언트 코드에 이미지를 두면 안 됨

### 3.2 마이페이지 - 지원한 프로젝트 보기 `[기능 5, 6번]`

#### 3.2.1 기본 규정
- 지원한 프로젝트의 **상태(지원/미팅/상세견적/계약중 등)는 없는 것으로 가정**
- 모든 지원 프로젝트를 리스트로 표시

#### 3.2.2 각 지원 목록 표시 항목
- 프로젝트 제목
- 견적
- 지원자 수
- 과업일수
- 상세보기 버튼

#### 3.2.3 상세보기
- 지원한 프로젝트의 **요약, 상세보기, 과업지시서는 생략**
- **"나의 지원서"** 버튼 클릭 시, 이전에 입력한 지원서 내용을 보여줘야 함

### 3.3 프로젝트 지원 - 프로젝트 검색 `[기능 1, 2번 (렌딩 페이지 공유)]`

#### 3.3.1 필터링 기능
| 필터 | 옵션 | 평가 포함 |
|------|------|-----------|
| 프로젝트 형태 | 전체 / 도급(원격) / 상주 | O |
| 참여파트 분류 | 기획, 디자인, 개발 등 (중복 선택) | **X (제외)** |
| 지역검색 | - | **X (제외)** |

#### 3.3.2 정렬 기능
- 프리모아 기본정렬
- 최신 등록 순
- 금액 높은 순
- 금액 낮은 순
- 마감 임박 순

#### 3.3.3 페이지네이션 `[기능 2번 핵심]`
- **페이지 사이즈 = 4**
- 새로운 페이지 요청 시마다 **서버로부터 데이터 수신** 필수

#### 3.3.4 핵심 제약 사항 ⚠️
> **프로젝트 형태 / 정렬 / 페이지 선택마다 새로운 데이터를 서버로부터 요청해야 함**
> **프론트엔드 단에서의 정렬은 인정하지 않음**

#### 3.3.5 프로젝트 요약 표시 항목
필수 표시:
- 제목
- 기간제 상주 / 도급 구분 표시
- 모집중 / 마감 상태
- 개발에 필요한 기술 스택
- 금액 정보 (기간제 상주는 **월 임금**, 도급은 **예상 비용**)
- 예상 기간
- 지원자 수
- 마감일정 (D-day 형태)

표시하지 않아도 되는 항목 (붉은색 박스 영역):
- 프로젝트 진행 방식 상세
- 계약방식 / 근무위치 / 근무형태 / 근무시간 / 근무기간 / PC 제공 여부
- 의뢰인 정보 (dks****, 연락처 인증 등)

### 3.4 프로젝트 상세보기

- 여러 탭(요약/업무내용/프리톡 등) 중 **요약 탭만** 보여주면 됨
- 표시 항목:
  - 고용형태 뱃지 (기간제 상주 / 도급), 모집 상태 뱃지
  - 등록일
  - 프로젝트명
  - 월 임금 또는 예상 비용
  - 예상 기간
  - 지원자 수
  - 마감일정 (D-day)
  - 관련 기술

---

### 3.5 프로젝트 지원하기 `[기능 7번]`

#### 3.5.1 공통 사항
- 요약 영역 표시 (모집 마감일, 예상 킥오프 일정, 고용형태, 프로젝트 분야, 진행 분류, 기획 상태, 미팅 희망 지역)
- **관련 포트폴리오 항목은 생략**
- **지원 내용에 이메일, 전화번호 등이 입력되었는지 검사하는 로직 필수 구현** ⚠️

#### 3.5.2 도급 프로젝트 지원 입력 항목
- 작업기간 (일)
- 지원 금액 (만원 단위, 프리모아 이용료 10% 포함)
- 지원 내용 (텍스트)

#### 3.5.3 상주 프로젝트 지원 입력 항목
- 기술구분 (개발자 / 디자이너 / 기획자 / 기타 포지션)
- 연차구분 (초급 1~5년 미만 / 중급 5~10년 미만 / 고급 10년 이상)
- 인원수
- 임금 (만원)
- 지원 내용

> **중요**: 도급의 경우 **인원수는 1명(본인)으로 한정**

#### 3.5.4 검사 항목
- 마이페이지에 정상 등록되는지 확인
- 렌딩페이지에서 해당 프로젝트의 **지원자 수가 증가**하는지 확인
- 도급/상주에 따라 입력 폼이 올바른지 확인
- 이메일/전화번호 입력 검증 로직 동작 확인

## 4. 의뢰인(Client) 요구사항

> **참고**: 의뢰인의 프로필 기능은 생략

### 4.1 프로젝트 의뢰하기 `[기능 8번]`

생성 형태(UI/플로우)는 팀에서 자유롭게 결정하되, 다음 항목들이 올바르게 등록되어야 함:

| 항목 | 입력 형태 |
|------|-----------|
| 프로젝트명 | 텍스트 |
| 모집마감일 | YYYY-MM-DD |
| 고용형태 | 도급외주 / 상주 |
| 예산 | 도급=제작예산, 상주=월급여 |
| 프로젝트 분야 | 텍스트 다중 선택 (개발, 디자인, 기획) |
| 기획상태 | 텍스트 |
| 미팅 희망 지역 | 텍스트 |
| 업무내용 | 텍스트 |
| 프로젝트 진행 방식 | 텍스트 |
| 필요 기술 스택 | 텍스트 다중 선택 |

**제외 항목**: 프로젝트의 현재 상황 / 예산 및 기간(중복) / 상세한 업무내용 / 참고사항

---

### 4.2 프로젝트 관리 `[기능 9, 10번]`

#### 4.2.1 등록한 프로젝트 현황 리스트
각 리스트 항목에 포함되어야 하는 정보:
- 프로젝트명
- 예상 금액
- 계약 형태 (도급 / 기간제 상주)
- 지원자 수
- 모집 마감일
- D-day
- 상세보기 버튼

#### 4.2.2 프로젝트 상세 화면 `[기능 9번 핵심]`
"상세" 클릭 시 다음 두 영역을 표시:

1. **의뢰 내용 요약**
   - 모집 마감일, 예상 킥오프 일정, 고용형태, 프로젝트 분야, 진행 분류, 기획 상태, 미팅 희망 지역

2. **지원자 리스트** (테이블 형태)
   - 순번 / 예상 금액 / 지원일(고용형태) / 상세보기 버튼
   - **"더보기" UI 필수 지원**
     - **페이지 사이즈 = 2**
     - 더 이상 볼 데이터가 없으면 **더보기 버튼이 없거나 비활성화** 되어야 함

#### 4.2.3 지원서 열람 `[기능 10번]`
- 지원자 리스트의 "상세보기" 클릭 시, 해당 지원자가 이전에 입력한 **지원서 내용 전체** 열람

## 5. 화면 흐름

### 5.1 렌딩 페이지
- 의뢰인과 개발자의 **렌딩 페이지는 동일 (프로젝트 찾기 화면)**
- 상단에 마이페이지 진입 메뉴 배치
- 각 역할(개발자/의뢰인)에 따라 마이페이지 화면을 다르게 표시

### 5.2 개발자 마이페이지
- **프로젝트 관리** 탭 + **프로필 관리** 탭만 표시
- (마이홈, 리뷰관리, 문서관리, 설정 등의 다른 탭은 구현 불필요)

### 5.3 의뢰자 마이페이지
- **프로젝트 의뢰하기** 탭 + **프로젝트 관리** 탭만 존재
- UI 구성은 팀에서 자유 결정

## 6. 평가 기준 (총 20점)

각 기능당 **2점**, 총 10개 기능 = 20점

### 6.1 렌딩페이지
| 번호 | 기능 | 핵심 검사 |
|------|------|-----------|
| **기능 1** | 프로젝트 형태/정렬 기준에 따라 프로젝트 표시 | 서버에서 필터링/정렬된 데이터 응답 |
| **기능 2** | 페이징 기능 | 페이지 사이즈 4, 페이지 요청마다 서버 호출 |

### 6.2 개발자
| 번호 | 기능 | 핵심 검사 |
|------|------|-----------|
| **기능 3** | 프로필 수정 - 기본정보 | 검색태그 x버튼 삭제, 5개 초과 알림, 나머지 필드 수정 가능 |
| **기능 4** | 프로필 수정 - 프로필 이미지 | 클라이언트→서버 이미지 전송, 서버 저장 후 경로 반환, 클라이언트가 경로로 갱신 (서버 로그 확인) |
| **기능 5** | 지원한 프로젝트 목록 보기 | 마이페이지에 지원 목록 표시 |
| **기능 6** | 상세보기 → 나의 지원서 클릭 → 지원서 내용 열람 | 이전 작성 내용 정확히 표시 |
| **기능 7** | 프로젝트 지원하기 | 이메일/전화번호 입력 검증 로직 **필수**, 도급/상주 폼 분기, 지원자 수 증가 |

> ※ PDF 23p에 "기능 5"가 두 번 표기되어 있으나, 24p 기준 기능 5는 "지원한 프로젝트 목록 보기"로 정정됨

### 6.3 의뢰인
| 번호 | 기능 | 핵심 검사 |
|------|------|-----------|
| **기능 8** | 의뢰 프로젝트 생성 | 모든 필수 필드 등록 |
| **기능 9** | 프로젝트 관리 - 의뢰내용 요약 + 지원자 리스트(더보기 UI, 페이지 사이즈 2) | 더보기 비활성화 처리 |
| **기능 10** | 지원자 선택 → 해당 지원서 보기 | 지원서 상세 열람 |

## 7. 더미 데이터 요구사항

### 7.1 필수 데이터
- **개발자 계정** (프로필 적절히 생성)
- **의뢰인 계정**
- **다양한 의뢰인의 프로젝트 의뢰 - 최소 10건 이상**
- 추가 검사 데이터는 팀에서 결정

### 7.2 데이터 다양성 권장
페이지네이션(4건/페이지), 정렬, 필터링 검증을 위해:
- 도급/상주 프로젝트가 골고루 섞여 있어야 함
- 다양한 금액대
- 다양한 등록일자 (최신순 정렬 검증)
- 다양한 마감일자 (마감임박순 정렬 검증)

## 8. 구현 제약사항 (⚠️ 감점 요소)

| 제약 | 위반 시 |

## 11. 주의해야 할 함정 정리

1. **이미지 처리**: 절대 클라이언트(public/assets 등)에 이미지를 두면 안 됨. 반드시 서버 로컬 폴더 → DB 경로 저장 → 클라이언트가 경로로 fetch
2. **정렬은 서버에서**: 프론트엔드 `array.sort()` 금지. 정렬 변경 시 API 재요청
3. **세션 인증 필수**: 습관적으로 JWT를 쓰지 말 것. Spring Security + HttpSession 또는 동등한 방식
4. **SPA 필수**: SSR 프레임워크(Next.js 기본 모드, Nuxt SSR 등) 사용 시 평가 불가. CSR 모드로 사용해야 함
5. **이메일/전화번호 차단 로직**: 지원 내용 텍스트에 정규식 검사 필수 (e.g. `\d{2,3}-\d{3,4}-\d{4}`, `\S+@\S+\.\S+`)
6. **더보기 페이지 사이즈 = 2**: 지원자 리스트에만 해당. 프로젝트 페이지네이션의 4와 헷갈리지 말 것
7. **도급의 인원수는 1명 고정**: 입력 폼에서 readonly 또는 강제 1
