# 요구사항 기준선

## 목표
- 발표자료 요구사항을 백엔드 구현 기준으로 다시 고정한다.
- 프론트 평가 항목까지 포함하되, 본 문서에서는 백엔드가 반드시 제공해야 하는 책임만 추린다.

## 확정된 전제
- 역할은 `개발자(DEVELOPER)`와 `의뢰인(CLIENT)` 두 가지다.
- 회원가입은 평가 필수 항목이 아니며, 검사 전 계정을 미리 준비해도 된다.
- 다만 테스트 편의를 위해 간단한 로그인/회원가입 API를 제공하는 것은 허용한다.
- 인증은 반드시 `세션 기반`이어야 하며 JWT는 사용하지 않는다.
- 랜딩 페이지는 `프로젝트 찾기` 화면이다.
- `지역검색`, `참여파트` 필터는 제외한다.
- `포트폴리오`, `과업지시서`, `의뢰인 프로필`은 제외한다.
- 개발자 지원 상태(`지원`, `미팅`, `상세견적`, `계약중`)는 없는 것으로 가정한다.

## 개발자 기능
- 프로필 수정
- 프로필 이미지 업로드
- 지원한 프로젝트 목록 조회
- 지원 프로젝트 상세 및 내 지원서 열람
- 프로젝트 검색 / 정렬 / 페이지 이동
- 프로젝트 지원서 작성 및 제출

## 의뢰인 기능
- 프로젝트 생성
- 등록한 프로젝트 목록 조회
- 프로젝트 상세에서 지원자 목록 조회
- 지원자 지원서 상세 조회

## 필수 판정 포인트
- 프로젝트 검색은 `프로젝트 형태`, `정렬`, `페이지`가 바뀔 때마다 서버에서 다시 조회해야 한다.
- 페이지 크기는 랜딩 목록 `4`, 지원자 더보기 목록 `2`로 고정한다.
- 개발자 프로필 이미지 업로드는 반드시 `multipart/form-data -> 서버 저장 -> DB 경로 저장` 순서를 따른다.
- 지원 내용에는 이메일 또는 전화번호가 포함되면 안 된다.
- 프로젝트 지원 후 개발자 목록과 의뢰인 상세 화면에서 지원자 수 증가가 확인되어야 한다.
- 더미 데이터는 최소 `프로젝트 10건 이상` 필요하다.

## 자료 불일치 처리
- 슬라이드 14는 실제 프리모아 폼 기준으로 해석한다.
- 도급 지원 폼은 `작업기간`, `지원금액`, `지원내용`을 사용한다.
- 상주 지원 폼은 `기술구분`, `연차구분`, `인원수`, `임금`, `지원내용`을 사용한다.
- 기능 번호 중복은 무시하고 기능 내용 자체를 기준으로 구현한다.
