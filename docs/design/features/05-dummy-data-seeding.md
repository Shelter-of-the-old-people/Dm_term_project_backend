# 더미 데이터 시딩 설계

## 목적
- 평가와 수동 테스트를 위해 최소한의 시연 데이터를 안정적으로 준비한다.

## 필수 데이터
- 개발자 계정 1개 이상
- 의뢰인 계정 1개 이상
- 프로젝트 10건 이상
- 도급/상주 프로젝트 혼합
- 지원 이력이 있는 프로젝트 포함

## 권장 데이터 구성
- 개발자 2명
- 의뢰인 2명
- 프로젝트 12~15건
- 마감 임박, 마감 완료, 최신 등록 프로젝트를 섞어서 정렬 테스트 가능하게 구성

## 시딩 전략
- 애플리케이션 기동 시 DB가 비어 있을 때만 insert
- `CommandLineRunner` 또는 SQL seed 사용
- H2 TCP 모드이므로 실제 데이터 파일은 `~/termdb.mv.db`에 유지되고, 재기동 시 중복 insert 방지 필요

## 예시 계정
- 개발자: `dev1 / 1234`
- 의뢰인: `client1 / 1234`

## 검증 포인트
- 랜딩 페이지 3페이지 이상 생성 여부
- 개발자 지원 목록 존재 여부
- 의뢰인 프로젝트 상세에서 더보기 가능 여부
- 이미지 경로는 더미 문자열이 아닌 실제 업로드 시나리오로 검증
